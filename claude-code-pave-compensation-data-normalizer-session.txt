⏺ Great lens — let's take the LLM question apart from the user's seat, because
  the answer falls out of where the friction actually is in the specialist's
  job.

  What the specialist actually does

  When a new company's file lands, the specialist has three cognitive tasks:

  1. Understand the raw data. What fields exist, what's in them, which are
  sparsely filled? In Acme's file, additional_compensation.commission_plan.ote
  exists on only 2 of 20 records — if you're eyeballing the first record, you'll
  never know it's there.
  2. Semantic matching. Decide that payRate, BASE_ANNUAL, and
  compensation.total_base_pay.amount all mean "base pay," ~18 times, choosing
  among 20–40 source paths per company.
  3. Catch the traps. payRate alone is a silently wrong mapping — two Globex
  employees are paid per Month, so without the annualize transform you'd ship
  $6,666.67 into salary benchmarks. Same class of trap: FULL_NAME is "Last,
  First", STATUS is coded A/T/L.

  Tasks 1 and 3-verification are solved deterministically, and must exist
  regardless of any LLM: a data profiler (field list, inferred types, fill
  rates, sample values) and a live validated preview (per-record
  ok/warning/error, raw-vs-normalized side by side). Those are the trust
  mechanisms. The specialist should never trust a mapping because software
  sounded confident — they trust it because they see 20 real records come out
  clean.

  Task 2 is pure drudgery, and it's a semantics problem, not a syntax problem.
  That's the LLM slot. Fuzzy string matching gets hireDate → hire_date but can't
  get supervisory_org → department, and it will never notice "payPer contains
  'Month' on 2 records, so payRate needs the annualize transform." An LLM
  reading field names plus sample values handles exactly that.

  The UX contract that makes it safe

  The critical product decision: the LLM never touches employee data at runtime 
  — it only drafts configuration, once, at onboarding. Draft → review → approve:

  - "Suggest mappings" prefills the mapping table. Every suggested row is
  visibly provisional (a "suggested" chip, with the evidence shown inline: the
  sample values it matched on), and uncertain fields say "needs review" instead
  of being silently filled.
  - Nothing runs until the human approves; the approved artifact is plain,
  inspectable config JSON. From then on the pipeline is deterministic — same
  config, same data, same output, every sync, auditable. For a comp-data company
  you can never say "the model decided your salary."
  - If the API key is missing, the tool degrades to fuzzy name-matching or plain
  manual mapping — full functionality, just slower. The dependency points the
  right way.

  Why this is worth building at all: reviewing a draft is cognitively much 
  cheaper than authoring from a blank canvas. Time-to-first-preview drops from
  ~15 minutes of hunting through 40 paths to ~1 minute of review-and-fix. And
  onboarding time per company is the metric the README's business context cares
  about ("onboard new customers without writing custom logic"). At
  interview-scale-question time, it compounds: with hundreds of saved configs,
  suggestions can be few-shot-primed from past configs for the same HR system.

  Where I would not put an LLM: inside the data pipeline as a runtime transform.
  That imports nondeterminism, cost, and failure handling into every sync. The
  one principled exception is Globex's free-text equityNotes ("500k RSU…
  Refresh: 200k RSU…") — genuinely unstructured, no deterministic parse exists.
  Even there the honest design is: keep the raw text in the schema, flag any
  extracted values as inferred, and treat full extraction as a
  stretch/discussion item rather than core.

  So my priority order under the 2-hour budget: profiler + mapping table +
  transforms + validated preview first (the tool must stand alone) → LLM suggest
  as leverage on top → equity extraction last or as DECISIONS.md discussion.

  Pitfall 1: the company's data structure changes later

  Say Globex switches payroll providers next quarter and the export renames
  payRate → basePayRate, adds a new equityValue column, and starts emitting a
  status "on_leave" we've never seen. Three distinct kinds of drift, and the
  engine can detect all three mechanically on any run, because the config
  declares exactly what it expects:

  - Missing mapped source — config references payRate, but it's absent from 100%
  of records → loud error per affected field, not silent nulls.
  - Unknown enum value — the status value-map covers active/terminated, sees
  on_leave → warning with the offending value and count ("2 records have
  unmapped status on_leave").
  - New unmapped source fields — equityValue appeared and no mapping uses it →
  surfaced in an "unmapped source fields" panel ("data you're currently dropping
  — intentional?").

  The product answer to "how do we update the config" is that the mapper isn't 
  just an onboarding tool — it's the ongoing maintenance surface. The specialist
  opens the company, the saved config loads, they point it at the new file, and
  the same live preview that powered onboarding now doubles as a drift report.
  They fix the two rows, re-preview, save. Config is data, not code: version
  bumps, updated_at stamps, no deploy, no engineer. (And this is where the LLM
  suggest feature earns its keep twice — "suggest a remap for the 2 broken
  fields" is the same mechanism applied to a diff.)

  Pitfall 2: mistakes in the data itself

  Layered validation, because different layers catch different failure classes:

  1. Parse-level — "N/A" where a number should be, an unparseable date → field
  issue.
  2. Sanity rules on the standard schema — this is the underrated one. A
  plausibility range on base_salary_annual (say $10k–$5M) catches a fat-fingered
  1650000... and it catches the forgotten annualize transform, because an
  unannualized monthly 6666.67 lands below the floor. The same rule set 
  validates both the data and the mapping. Similarly: bonus_target_pct in 0–100,
  termination_date ≥ hire_date, status = terminated ⇒ termination date
  expected.
  3. Cross-record — duplicate employee_ids within a run.

  Then disposition, which is a product decision: errors quarantine the record
  (excluded from clean output, listed with reasons), warnings include it 
  flagged, and the summary bar always reconciles ("17 clean, 2 warnings, 1
  quarantined — of 20"). Nothing silently dropped, nothing silently nulled.
  Clicking any flagged record shows raw vs. normalized side by side so the
  specialist can tell in seconds whether it's a data problem (kick back to the
  customer) or a mapping problem (fix in the mapper).

  Last piece: lineage. Every normalized output is stamped with {config version, 
  source file, run timestamp}. When a bad batch surfaces downstream, you can
  trace it to the exact config version and input — and because the engine is
  deterministic, reruns reproduce exactly.

  Scope honesty for the 2-hour build: drift detection (all three classes), parse
  + sanity + cross-field validation, quarantine/warning disposition, and config
  version stamping are all cheap because they live in the engine I'm building
  anyway — they're in. Version history UI, config diffs, automated drift alerts
  on scheduled syncs: discussed in DECISIONS.md, not built.

  Two design principles this encodes (both good DECISIONS.md material):

  1. Don't conflate distinct concepts, even when only one source distinguishes 
  them. You spotted exactly this: job_family ("Engineering" as a discipline) is
  not department ("Customer Success" dept contains Solutions Engineers), and
  supervisory_org is really department + team in one string — hence the split
  transform. Notably, Globex's hierarchy runs the opposite direction: its
  division ("Engineering") is broader than its department ("Data & Analytics") —
  the reverse nesting of Acme. Separate optional fields let each source land
  honestly; a specialist may choose to also map department → job_family for
  sources without a family, but the tool never conflates silently.
  Sparse-but-honest beats dense-but-wrong for benchmarking.

  level. Acme's P4 isn't another company's L4. Production Pave solves this with
  company-level → standard-level mapping — which is exactly our value-map
  transform applied at a different layer, a nice "how does this scale" answer
  for the interview. Out of scope to build, one line to design for. Same story

❯ Are we going to store each entity as one or should we split it into several   
  entities/tables? Pros/cons?                                                   

⏺ Good question — and it's worth separating two things that often get conflated:
  the logical record shape (what the engine emits — the contract shared by
  config, UI, preview, tests) and the physical storage (how we persist it). The
  comp-shape question we've been circling is the logical one; your question is
  the physical one, and it has a fairly crisp answer for a pipeline product like
  this.

  The two options

  A. One entity: an employee record document (equity grants embedded), grouped 
  under a per-run dataset.

  B. Split entities: employees, equity_grants (FK employee_id), optionally 
  comp_components, runs — relational style.

  Fit to the pipeline
  A: Single document per employee: Natural: 1 raw record → 1 normalized record,
  atomic; no assembly
  B: Split tables: Engine output must be disassembled on write, re-joined on
  read
  ────────────────────────────────────────
  Preview/demo UI
  A: Single document per employee: One fetch renders the table
  B: Split tables: Joins for every render
  ────────────────────────────────────────
  Grant-level analytics ("avg refresh size by level")
  A: Single document per employee: Requires unnesting the embedded array
  B: Split tables: Direct SQL over equity_grants
  ────────────────────────────────────────
  Normal-form hygiene
  A: Single document per employee: Repeating group embedded (1NF violation,
  technically)
  B: Split tables: Clean 1NF, FKs, integrity
  ────────────────────────────────────────
  Mutation/updates
  A: Single document per employee: Irrelevant — records are immutable pipeline 
  outputs, never edited in place
  B: Split tables: Update-friendliness we don't need
  ────────────────────────────────────────
  Comp history / effective dating later
  A: Single document per employee: Snapshot per run; history = append runs
  B: Split tables: Extends naturally to effective-dated component rows
  ────────────────────────────────────────
  2h cost

  My recommendation

  Option A — with the honest footnote that it's really three entities already:

  1. configs — one document per company, with version / updated_at (its own
  lifecycle, edited by the mapper).
  2. datasets (runs) — the envelope: {company, source_file, source_generated_at,
  config_version, run_at, summary}.
  3. employee_records — one document per employee per run, grants embedded, each
  carrying its issues[].

  The reasoning that makes this deliberate rather than lazy: normalized records
  are snapshot facts about a sync run, not mutable entities. Run-scoped
  immutable documents buy reproducibility (same config + file → identical
  dataset), diffability between runs (that's literally the drift-detection
  feature), and audit lineage — the three things we said matter when data goes
  bad. Classic relational decomposition optimizes for in-place updates and
  referential integrity, which this pipeline never does.

  The embedded equity_grants array is the right call by the same test we used
  for the schema shape: grants have no independent identity in any source (even
  Globex's two grants live inside one text field), they're bounded (1–3 per
  person), and this tool never queries grants standalone. The moment downstream
  analytics needs grant-level SQL, you split them out in the warehouse layer
  (dbt-style: fact_equity_grant unnested from the documents) — the document
  schema and the mapper don't change at all. That's the production evolution
  story for the interview: mapper → normalized documents landed per run →
  warehouse models split/flatten as analytics demands, plus SCD-style employee
  identity across runs. Building relational decomposition inside the 2-hour tool
  would be paying that cost at the wrong layer.

  For the exercise's persistence this means plain JSON files:
  data/configs/<company>.json, and normalized output as {dataset envelope, 
  records[]} — computed live for preview, written to disk on "export."
